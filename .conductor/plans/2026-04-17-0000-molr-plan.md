## MoLR Implementation Plan

### Overview

The system replaces each routed expert with a Mixture of Low-Rank (MoLR) approximator consisting of K components, each a low-rank SwiGLU MLP, combined via a learned soft router. A learned scalar confidence head predicts approximation error from the input, enabling optional fallback to the full expert. All K components are always evaluated at inference time for this implementation.

---

### Module 1: Offline Preparation

This module runs once on the original model weights before any training occurs. No GPU is required beyond what's needed to load and process the weights.

**1.1 Weight Extraction and SVD**

For each of the experts across MoE layers, extract the three weight matrices and compute full SVD. Store the singular value spectra alongside the factorizations. The spectra are used in two ways: to choose rank r per expert, and to diagnose approximation quality later.

The rank selection criterion should be based on explained variance — choose r such that the top-r singular values capture at least some target fraction (e.g. 90%) of the total squared singular value sum. This will produce different ranks for different experts and different matrices within the same expert. Experts with fast spectral decay (most variance in a few directions) can use very low rank; experts with flat spectra need higher rank. Store the chosen rank per expert per matrix.

**1.2 Strided Spectral Initialization of Components**

For K components, assign singular vectors using a strided pattern: component i receives singular vectors at positions i, i+K, i+2K, and so on. This distributes high- and low-importance directions evenly across components so that all components start with approximately equal reconstruction quality, avoiding initialization bias toward component 0.

After assignment, normalize each component's factors so that all K components have equal Frobenius norm. This prevents the load balancing problem at initialization where the router trivially learns to always pick the component with the largest singular values.

**1.3 Covariance Estimation for Synthetic Data**

Run a small forward pass through the full model on a modest corpus sample — on the order of 10,000–50,000 tokens is sufficient. For each expert, record the inputs that were routed to it and compute the sample mean and covariance of those inputs. Store these per expert as (mu, L) where L is the Cholesky factor of the covariance matrix, used in training to generate spectrally shaped synthetic inputs.

This is the only step that requires running the full model on real data. Everything downstream uses synthetic data generated from this estimated distribution.

---

### Module 2: MoLR Architecture

**2.1 Component Structure**

Each component is a low-rank SwiGLU MLP defined by six parameter matrices:

- gate_A (d_model × r), gate_B (r × d_intermediate)
- up_A (d_model × r), up_B (r × d_intermediate)
- down_A (d_intermediate × r), down_B (r × d_model)

The forward pass computes gate and up projections through their respective two-stage GEMVs, applies SiLU and elementwise multiply in the bottleneck, then projects back through down. The two-stage structure means the largest intermediate tensor is of dimension r, not d_intermediate, so peak memory per component forward is small.

**2.2 Router**

A single linear layer from d_model to K, followed by softmax. No hidden layer. Produces K weights that sum to 1. These weights are used to compute the output as a weighted sum of all K component outputs.

**2.3 Error Prediction Head**

A single linear layer from d_model to 1, followed by Softplus to ensure the predicted error is non-negative. Takes the raw input x and produces a scalar predicted approximation error. This shares no parameters with the router and is trained jointly with the rest of the MoLR.

**2.4 Full Forward Pass**

1. Compute router weights from x (K-dimensional softmax)
2. Compute all K component outputs (K independent SwiGLU forwards)
3. Compute weighted sum of component outputs
4. Compute predicted error from x via error head
5. Return (output, predicted\_error)

At inference when fallback is not needed, the predicted error can be ignored. When the fallback mechanism is active, the predicted error is checked before any component computation — since the error head is just a single linear layer, it costs essentially nothing and the fallback decision can be made before loading component weights.

### Module 3: Training

**3.1 Synthetic Data Generation**

At training time, generate input batches by sampling from N(mu, LL^T) using the per-expert covariance estimated in Module 1. Concretely: sample standard Gaussian vectors z, multiply by L^T, add mu. Each batch consists entirely of synthetic inputs; no real text data is used during training.

The target outputs are computed by running each synthetic input through the original (frozen) expert weights. This is cheap since each expert is a simple SwiGLU MLP — just three matrix multiplies and an elementwise operation.

**3.2 Loss Function**

Three terms, summed:

**Output MSE.** The primary objective. Mean squared error between the MoLR output and the original expert output on the same input.

**Load balancing.** Penalizes unequal average routing across components. Computed as the sum of squared mean gate weights across the batch — minimized when all components receive equal average weight. Weighted by a small coefficient (e.g. 0.01) to avoid overwhelming the primary loss.

**Error head supervision.** MSE between the predicted scalar error and the true approximation error ‖MoLR(x) - Expert(x)‖₂, where both quantities are computed during the same forward pass. The true error must be detached from the computation graph so that the primary network is not penalized for making the error head's job easier by increasing its own output error. Weight this term in the range 0.05–0.1.

**3.3 Training Procedure**

Freeze all original expert weights. Train only the MoLR parameters: all component factors, router weights, and error head weights. Use Adam with a moderate learning rate (around 1e-4). Batch sizes of 256–1024 work well since each sample is a single vector and the forward pass is cheap.

Train each expert independently. This is embarrassingly parallel — all experts can be trained simultaneously if GPU memory permits, or in large parallel batches. Each expert requires only its own weight matrices and the estimated covariance; there is no dependency between experts.

Training is complete when output MSE on a held-out synthetic validation set has converged. In practice this is likely 10,000–50,000 steps per expert depending on K and r. Monitor the load balancing diagnostic (router entropy averaged across validation inputs) to confirm components have specialized rather than collapsed.

**3.4 Per-Expert Quality Validation**

After training, evaluate each MoLR on its held-out validation set and record:

- Mean cosine similarity between MoLR output and true expert output
- Mean relative output norm error
- Router entropy (higher = more balanced specialization)
- Correlation between predicted error and true error (Pearson r)

Flag experts where cosine similarity falls below a threshold (e.g. 0.95) or where error prediction correlation is weak (r < 0.7). These experts are candidates for higher rank, higher K, or inclusion in a small RAM-resident cache of full expert weights as permanent fallbacks.

---

### Module 4: Storage and Memory Layout

**4.1 Optional Full Expert Cache**

Maintain a small RAM-resident cache of full expert weights for experts flagged as poor approximations during validation (Module 3.4) and for experts that frequently trigger the fallback mechanism at runtime. Allocate a fixed RAM budget for this cache (e.g. 4 GB, covering ~200 full experts) and populate it with the worst-approximated high-frequency experts.

---

### Module 5: Inference Integration

**5.1 Per-Expert Forward with Optional Fallback**

The inference path for each routed expert call:

1. Run error head on x (on GPU, negligible cost — router and error head weights are VRAM-resident)
2. If predicted error exceeds threshold: skip MoLR, load full expert from cache or SSD, run full forward, return
3. Otherwise: load MoLR factors from RAM or SSD, run all K component forwards, compute weighted sum, return

The threshold is a runtime parameter, not a trained one. It is calibrated offline by sweeping values on a validation set and measuring the resulting (fallback rate, output quality) tradeoff. Store this as a lookup table mapping quality targets to thresholds, so users can configure a simple quality level without understanding the internals.

**5.2 Fallback Rate Monitoring**

Track per-expert fallback rates during inference. Experts with consistently high fallback rates at a given threshold are candidates for promotion to the full expert RAM cache, or for retraining with higher K or r. This creates a feedback loop where real inference workloads inform future model optimization without requiring labeled data.

**5.3 Interaction with Original MoE Router**

The MoLR operates transparently as a drop-in replacement for the original expert forward function. The model's own routing mechanism (which selects which experts to activate per token) is unchanged. MoLR only replaces what happens *inside* each selected expert, not which experts are selected.

---

### Summary of Key Parameters

| Parameter | Recommended Starting Point | Effect of Increasing |
|---|---|---|
| K (components) | 4 | Better quality, more compute and RAM |
| r (rank per component) | 64 | Better quality, more RAM |
| Covariance sample size | 10K–50K tokens | Better synthetic data distribution |
| Training steps per expert | 20K | Better convergence |
| Load balance coefficient | 0.01 | Stronger specialization pressure |
| Error head coefficient | 0.05 | Better fallback calibration |
| Full expert cache size | ~200 experts (4 GB) | Fewer SSD fallback reads |
